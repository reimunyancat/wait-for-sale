# WaitForSale - Steam 게임 할인 예측 서비스

## 📋 프로젝트 개요

### 프로젝트 설명

Steam에 있는 게임들의 할인 패턴을 분석하여, 딥러닝을 통해 할인이 예상되는 게임들을 예측하고 보여주는 웹 서비스

### 프로젝트 목적

원하는 게임이 있을떄 고민하다가 구매하였으나 얼마 안가서 그 게임이 할인을 하면 느끼는 허탈감등을 해결하기 위함이다
반대로 원하는 게임이 있을때 할인할때까지 무기한 기다리기에는 과연 이 게임이 언제 할인할까라는 고민만 계속 안고 가기 때문에 그 고민을 어느 정도 해소하기 위함이다

### 타겟 사용자

Steam에서 게임을 찾고있는 사람들

## 🎯 핵심 기능

### 필수 기능

1. **게임 할인 정보 수집**
   - Steam 게임들의 과거 할인 이력 데이터 수집

   - 할인 예측 결과 시각화

## 📈 개발 단계 (업데이트: 2025-12-06)

### 🧠 모델 개선 및 API 서버 구축 탐험일지 (2025-12-06)

#### **1차 시도: 단일 게임 모델 (실패)**

- **결론:** 단일 게임 데이터만으로는 일반화된 할인 패턴을 학습하기에 한계가 명확.

#### **2차 시도: "프로파일러 모델" (성공적인 기반 확보)**

- **결과:** **테스트 정확도 약 70%** 달성. 새로운 접근법의 유효성을 성공적으로 검증.

#### **3차 시도: 모델 고도화 및 파이프라인 재구축 (초기 성공)**

- **전략:**
  - **대규모 특징 공학:** '스팀 주요 세일 기간' 뿐만 아니라, `openapi.json` 분석을 통해 발굴한 **'개발사', '배급사', '출시 후 경과일'** 등 추가 메타데이터를 모두 특징으로 구현.
  - **데이터 확장:** 학습 게임 수를 7개에서 24개로 대폭 늘려 모델의 일반화 성능 강화.
  - **하이퍼파라미터 최적화:** `Optuna`를 도입하여 100회의 테스트를 통해 최적의 하이퍼파라미터 조합을 자동으로 탐색. DB에 저장하여 최적화 과정을 이어갈 수 있도록 구현.
  - **코드 리팩토링:** 유지보수성 및 확장성을 위해 기존의 단일 코드를 `train.py`, `model.py`, `data.py`로 기능에 맞게 분리.
  - **안정성 강화:** API 통신 재시도 로직 및 메모리 오류를 회피하는 '가지치기' 기능을 구현하여 파이프라인 안정성 확보.
- **결과:** Optuna를 통해 검증 데이터(Validation set) 기준 F1 Score **0.85**의 최고점을 기록하는 하이퍼파라미터 조합 발굴.

#### **4차 시도: 과적합 해결 및 API 서버 기반 구축 (진정한 성공)**

- **문제 식별:** 3차 시도에서 찾은 최적의 하이퍼파라미터로 최종 모델을 학습시켰으나, 테스트 데이터에서 F1 점수 **0.35**라는 매우 낮은 성능을 기록. 이는 검증 데이터셋에 대한 **과적합(Overfitting)**이 심각하게 발생했음을 의미.
- **해결 전략:**
  - **조기 종료(Early Stopping) 구현:** 훈련 중 검증 데이터(Validation Set)의 F1 점수를 지속적으로 모니터링하여, 점수가 일정 기간(patience=10) 이상 향상되지 않으면 가장 성능이 좋았던 시점의 모델을 선택하고 학습을 중단하는 로직을 `train.py`에 구현.
  - **추론을 위한 전처리기 저장:** 예측 API 서버에서 훈련 시와 동일한 데이터 전처리(스케일링, ID 매핑)를 수행할 수 있도록, `MinMaxScaler` 객체와 `developer/publisher` ID 맵을 모델 파일(`.pth`)에 함께 저장하도록 `data.py` 수정.
- **결과:**
  - 조기 종료 기법이 성공적으로 과적합을 방지하여, 최종 테스트 F1 점수가 **0.85**로 대폭 향상됨.
  - 추론에 필요한 모든 정보(모델 가중치, 하이퍼파라미터, 전처리기)가 포함된 `trained_model.pth` 파일 저장 완료.

#### **5차 시도: 예측 API 서버 완성 및 의존성 지옥 탈출 (진정한 최종 성공)**

- **1차 문제 (데이터 분할 오류):** 4차 시도에서 성공한 모델을 재학습시켰으나, 테스트 F1 점수가 **0.13**으로 급락하는 현상 발생. 원인 분석 결과, `data.py`에서 데이터를 게임별로 순차적으로 분할하여, 모델이 특정 게임에만 과적합되고 다른 게임에는 일반화되지 못하는 **데이터 오염(Data Contamination)** 문제임을 확인.
- **해결:** `data.py`에서 훈련/검증/테스트 세트를 나누기 전에, 전체 시퀀스 데이터를 무작위로 섞어주는(`np.random.permutation`) 로직을 추가.
- **결과:** 데이터 분할 문제를 해결하고 재학습한 결과, 최종 테스트 F1 점수가 **0.88**로 크게 향상되며 모델의 일반화 성능을 성공적으로 확보.

- **2차 문제 (API 서버 실행 오류):** 완성된 모델을 서빙하기 위해 `uvicorn`으로 FastAPI 서버(`server.py`)를 실행했으나, 수많은 에러와 함께 실패.
- **해결 과정 (의존성 지옥 탈출기):**
    1.  **`ModuleNotFoundError` 해결:** 가상환경(`venv`)이 아닌 시스템의 파이썬으로 서버가 실행되어 `pandas` 등의 라이브러리를 찾지 못하는 문제. `uvicorn` 대신 `python -m uvicorn`으로 실행하여 가상환경의 인터프리터를 명시적으로 사용하도록 수정. 또한, `requirements.txt`에 누락되었던 `fastapi`, `uvicorn`을 추가하고, PyTorch와의 호환성을 위해 `numpy<2.0` 버전을 명시하여 환경 의존성 문제 해결.
    2.  **`ImportError` 해결:** 프로젝트 루트에서 서버를 실행할 때, `server.py` 내부의 `from model import ...` 같은 절대 경로 방식의 임포트가 문제를 일으킴. `from .model import ...` 처럼 **상대 경로 임포트**로 수정하여 해결.
    3.  **`FileNotFoundError` 해결:** `trained_model.pth`, `games_list.json` 파일을 찾지 못하는 문제. 스크립트 실행 위치(CWD)에 따라 상대 경로가 달라지는 문제임을 파악. `pathlib` 모듈을 사용하여 파일의 위치를 기준으로 **절대 경로를 생성**하도록 모든 파일 로딩 로직을 수정하여 해결.
    4.  **PyTorch `weights_only` 오류 해결:** PyTorch 2.6부터 보안 강화로 인해 `torch.load`의 기본값이 `weights_only=True`로 변경되어, `scaler` 등 추가 정보가 포함된 모델 파일 로드에 실패. `torch.load(..., weights_only=False)` 옵션을 추가하여 **직접 만든 안전한 파일임을 명시**하고 해결.
    5.  **`.env` 경로 문제 최종 해결:** 사용자의 날카로운 지적으로 `backend` 코드 또한 `model` 코드와 동일한 `.env` 경로 문제를 겪을 것임을 파악. `.env` 파일의 위치를 **프로젝트 최상단으로 표준화**하고, `itad_api.py` (Python)와 `index.ts` 등 (Node.js) 양쪽 모두 `pathlib`, `path` 모듈을 사용해 프로젝트 최상단 폴더를 기준으로 `.env`를 찾도록 로직을 **리팩토링**하여 아키텍처의 견고성 대폭 향상.
- **최종 결과:**
  - 수많은 디버깅 끝에, 예측 모델을 성공적으로 서빙하는 FastAPI 서버가 완벽하게 동작함을 확인 (`200 OK`).
  - 이로써 **Phase 2의 모든 목표가 완료됨.**

### 💻 프론트엔드 & 백엔드 API 개발 탐험일지 (2025-12-07)

Phase 2에서 완성된 모델 API를 사용자에게 보여주기 위한 웹 인터페이스 개발을 시작.

#### **1. 프론트엔드 초기 설정 및 기술 선택**

- **Vite 채택 이유:** 초기 세팅의 속도와 개발 경험을 극대화하기 위해 `Create React App` 대신 `Vite`를 선택. Vite는 네이티브 ES 모듈을 기반으로 하여 개발 서버의 구동 속도와 HMR(Hot Module Replacement) 속도가 압도적으로 빠르다. 이는 UI를 실시간으로 확인하며 빠르게 수정해야 하는 프론트엔드 개발 과정에서 생산성을 크게 향상시킨다.
- **Tailwind CSS 채택 이유:** 신속한 프로토타이핑과 일관된 디자인 시스템 유지를 위해 유틸리티 우선(Utility-first) 프레임워크인 `Tailwind CSS`를 선택. `Styled-Components`나 `Emotion` 같은 CSS-in-JS 방식은 특정 컴포넌트에 스타일이 종속되는 장점이 있지만, 이번 프로젝트처럼 전체적인 프로토타입을 빠르게 구축하는 데에는 미리 정의된 유틸리티 클래스를 조합하는 Tailwind가 더 효율적이라고 판단.
- **초기 설정 과정의 난관 (의존성 및 명령어 오류):**
  1.  `npx tailwindcss init -p` 명령어 실행 시, `could not determine executable to run` 에러가 반복적으로 발생.
  2.  `npm exec`, 실행 파일 직접 경로 호출(`.\node_modules\.bin\...`) 등 다양한 시도를 했으나 모두 실패. 이는 로컬 `node_modules`의 문제가 아닌, 사용자의 시스템 환경 자체의 문제(Path, npm 캐시 등)로 강력하게 추정되었다.
  3.  **해결:** 명령어 실행을 통한 자동 설정을 포기하고, `tailwind.config.js`, `postcss.config.js` 파일을 **수동으로 직접 생성**하는 방식으로 문제를 우회하여 해결.
  4.  이후에도 `@import`, `@tailwind` 등 CSS 지시어가 Vite/PostCSS에 의해 잘못 해석되는 현상이 간헐적으로 발생. `@import`를 `index.html`의 `<link>` 태그로 대체하고, `@apply` 사용을 최소화하는 등 **더 안정적이고 표준적인 방식으로 코드를 수정**하여 안정성을 확보.

#### **2. 백엔드 API 개발 및 연동**

- **검색 API 구현:** 사용자가 게임 이름을 검색할 수 있도록 `GET /api/games/search` 엔드포인트를 `Express`의 라우터로 구현. 데이터베이스 로직(`db/index.ts`)과 API 로직(`routes/games.ts`)을 분리하여 코드의 역할을 명확히 했다.
- **데이터베이스 연결 문제 해결 (디버깅 대장정):**
  1.  **문제 식별:** 백엔드 서버 실행 시, `.env` 파일이 올바르게 존재함에도 불구하고 `SASL: SCRAM-SERVER-FIRST-MESSAGE: client password must be a string` 에러가 발생하며 데이터베이스 연결에 실패.
  2.  **가설 1 (실패):** `.env` 파일 내 비밀번호 값의 따옴표 문제로 추정했으나, 사용자의 확인 결과 따옴표가 없음에도 문제가 지속됨.
  3.  **가설 2 (성공적 접근):** `dotenv` 로그의 `injecting env (0)` 메시지를 통해, `.env` 파일을 찾았으나 변수를 제대로 읽어오지 못하는 **경로 문제**임을 파악. `.env` 파일을 임시로 `backend` 폴더 내로 옮기고, `dotenv.config()` 경로를 비워 로컬 파일을 읽게 하자 `injecting env (7)`로 로그가 바뀌며 변수를 읽는 데 성공함.
  4.  **궁극적 해결책 (리팩토링):** `process.env` 자체의 신뢰성에 문제가 있다고 판단, `dotenv`가 파싱한 객체를 `connectDb` 함수에 **직접 인자로 전달**하는 방식으로 코드를 리팩토링하여 문제를 원천적으로 해결. 이 방식은 환경 변수 충돌 가능성을 완전히 배제하는 가장 견고한 방법이다.
  5.  **진정한 원인 발견 (사용자의 통찰):** 사용자의 지적으로, `dotenv`의 경로 설정이 `../../.env`가 아닌 `../../../.env`로 잘못 계산되어 있었음을 최종 확인.
  6.  **코드 복원:** 원인이 명확해졌으므로, 사용자의 요청에 따라 복잡했던 '직접 값 전달' 코드를 다시 **원래의 간결한 `process.env` 사용 코드로 되돌렸다.** 이로써 아키텍처의 간결함을 유지하면서 문제를 해결할 수 있었다.

#### **3. 사용자 피드백 기반 프론트엔드 고도화**

- **1차 피드백 반영 (컨셉 변경):** "단순 검색 페이지는 현실성이 떨어진다"는 사용자의 핵심적인 UX 피드백을 수용. 단순히 검색 결과만 보여주는 대신, **'곧 할인 예측', '인기 게임' 등을 먼저 보여주는 '대시보드' 형태**로 기획을 전면 수정. 이를 위해 `GameCard`, `GameList` 등 재사용 가능한 컴포넌트 아키텍처를 설계.
- **2차 피드백 반영 (디자인 폴리싱):**
  - **심미성:** "디자인이 2000년대 같다"는 피드백을 반영, **Steam과 유사한 다크 블루/차콜 색상 팔레트와 세련된 'Inter' 웹 폰트를 도입**하여 전체적인 룩앤필을 현대적으로 개선.
  - **레이아웃:** "카드가 너무 크고 글씨가 겹친다"는 문제를 해결하기 위해, 카드 내부 디자인을 재설계하고 그리드 시스템을 더 촘촘하게 수정하여 정보 밀도를 높임.
  - **기능:** "할인율 표시가 구체적이지 않다"는 지적에 따라, **'원래 가격'에 취소선을 긋고 '할인된 가격'을 함께 표시**하는 방식으로 UI를 수정하여 사용자가 할인 정보를 명확히 인지할 수 있도록 개선.

#### **4. 현재 상태 및 다음 단계**

- 기능적으로 동작하는 핵심 백엔드 API와, '대시보드' 컨셉의 프론트엔드 프로토타입이 완성됨.
- **다음 단계:** 사용자가 직접 제작한 **Figma 디자인 시안**을 기반으로, 현재의 프로토타입을 프로덕션 수준의 정교한 UI/UX로 최종 구현할 예정.
